# ADR-0034: Accessibility, WCAG 2.2 AA Minimum

- **Status:** Accepted
- **Date:** 2026-04-27
- **Author:** AJ Quick
- **Related:** [ADR-0035](0035-dense-data-readability.md), `src/lamella/static/app.css`, `src/lamella/static/components.css`, `src/lamella/templates/_components/`

## Context

Lamella is a self-hosted, single-user application. "Single-user" does not
exempt it from accessibility requirements. The user may work on a high-DPI
monitor one day and a low-contrast laptop screen the next. They may use a
keyboard-only workflow during a focused accounting session. They may eventually
hand off access to a bookkeeper whose needs differ from their own.

More practically: the accessibility requirements that WCAG 2.2 AA defines,
sufficient contrast, keyboard reach, visible focus, semantic HTML, also catch
bugs. A `<div onclick>` instead of a `<button>` is both an accessibility
failure and a keyboard navigation failure for sighted users. Missing labels
break screen readers and break autofill heuristics. Color-only error states
are invisible to colorblind users and to users viewing the app on a washed-out
screen.

The codebase uses vanilla CSS (`src/lamella/static/app.css`,
`src/lamella/static/components.css`) with no CSS framework. Semantic HTML is
possible without fighting a framework's defaults. ARIA attributes are rarely
needed because the semantic elements already carry the right roles.

The Phase 7 violation scan found 7 violations: 2 div-onclick patterns, 3
`outline: none` without focus-visible replacement, plus other minor issues.

## Decision

Every user-facing surface in Lamella meets WCAG 2.2 AA. This is not a goal
for a future release. It is a constraint on every change merged today.

Specific obligations:

1. **Contrast.** Text against background: ≥ 4.5:1 for normal text, ≥ 3:1 for
   large text (≥ 18pt or ≥ 14pt bold). UI component boundaries (button
   borders, input outlines, focus rings): ≥ 3:1 against adjacent color.
   Measure with a contrast checker before committing a new color token.
2. **Keyboard navigation.** Every interactive element is reachable via
   Tab / Shift-Tab. Tab order matches the visual reading order. Enter
   activates buttons and submits forms. Escape closes modals and popovers.
3. **Visible focus indicator.** Every interactive element shows a visible
   focus indicator when focused by keyboard. `outline: none` without a
   replacement indicator is forbidden. The replacement (box-shadow ring,
   border change, etc.) must meet the ≥ 3:1 contrast ratio against adjacent
   color.
4. **Semantic HTML.** `<button>` for buttons. `<a>` for navigation (changes
   the URL). `<form>` for submission. `<table>` for tabular data; not for
   layout. `<nav>`, `<main>`, `<header>`, `<footer>` as landmarks.
   `<div onclick>` and `<span onclick>` are forbidden for interactive roles.
5. **Labels.** Every `<input>`, `<select>`, and `<textarea>` has an associated
   label via `<label for="id">` or `aria-label`. Placeholder text is not a
   label substitute. It disappears on input and fails for screen readers.
6. **ARIA only where semantic HTML cannot.** `aria-live="polite"` for dynamic
   regions that update without a page reload (job progress, inline errors).
   `aria-expanded` on disclosure buttons. `aria-label` on icon-only buttons
   (the `icon_btn` macro in `buttons.html` already handles this). No ARIA
   role that duplicates what a semantic element already provides.
7. **Color is not the sole carrier of meaning.** Success, error, and warning
   states MUST carry a text label or icon in addition to color. A green border
   alone is not enough; "Saved" text or a checkmark icon must accompany it.

## Consequences

### Positive
- Keyboard-only workflows work. A user doing heads-down ledger review can tab
  through the review queue, accept or dismiss, and move to the next item
  without touching the mouse.
- Semantic HTML is easier to test. `axe-core` and Lighthouse catch structural
  violations automatically; color contrast is the main manual check.
- The `_components/` macro layer already handles `aria-label` on icon-only
  buttons (`buttons.html::icon_btn`, `_txn_actions.html` action buttons).
  Compliance is enforced by the macro. Callers cannot forget it.

### Negative / Costs
- Color choices for the existing palette must be verified. Some token pairs
  may need adjustment to meet 4.5:1. A one-time audit of `app.css` color
  tokens is required.
- Every new template must be checked during PR review. Axe-core in CI catches
  structural issues, not all contrast issues (contrast depends on computed
  styles which require a rendered browser).

### Mitigations
- Lighthouse / axe-core scan in CI targets zero accessibility issues on the
  main surfaces. CI failure blocks merge.
- A manual keyboard-only walkthrough of the review queue, card view, and
  search page runs quarterly. Results are logged in a tracking issue.
- The `icon_btn` macro and `_txn_actions.html` patterns show how to write
  accessible icon buttons. New contributors copy the pattern; they do not
  invent it.

## Compliance

- **Automated:** Lighthouse accessibility score ≥ 95 on `/`, `/review/staged`,
  `/search` in CI. Axe-core zero-violation gate on those same pages.
- **Grep violations:**
  - `grep -rn "<div[^>]*onclick" src/lamella/templates/`. Each match is a
    semantic HTML violation.
  - `grep -rn "outline: none" src/lamella/static/`. Each match must have an
    adjacent replacement focus indicator or it is a violation.
  - `grep -rn "<input[^>]*>" src/lamella/templates/` where no preceding
    `<label` or `aria-label` exists on the same element. Missing label.
- **Manual:** Quarterly keyboard-only walkthrough. Tab order, Enter/Escape
  behavior, visible focus on every interactive element.
